
# AI 管理平台统一响应协议设计（兼容 OpenAI + Agent 扩展）

## 1. 设计目标

本协议用于统一 AI 管理平台的会话响应格式，兼顾以下目标：

1. 对外尽量兼容 OpenAI Chat Completions 协议，方便复用现有前端客户端、SDK、网关与工具生态
2. 对内保留 Agent / Workflow / MCP / Tool 的执行过程表达能力
3. 支持非流式与流式两种模式，并保证语义基本一致
4. 支持前端展示增强信息，例如：
   - 执行步骤
   - 推荐追问
   - 建议操作按钮
   - 确认节点
   - 来源信息
   - 业务状态
5. 允许普通 OpenAI 客户端在**非流式**下忽略顶层扩展字段时仍可正常使用；**流式**下见 §2.4（部分客户端会校验每一条 SSE 行，不能与扩展混用）

---

## 2. 总体原则

### 2.1 标准主干 + 私有扩展

协议分为两层：

- **标准层**：尽量对齐 OpenAI Chat Completions
- **扩展层**：使用 `x_agent` 承载平台自定义能力

即：

- 标准客户端只读取：
  - `id`
  - `object`
  - `choices`
  - `message`
  - `content`
  - `finish_reason`
- 平台自有前端额外读取：
  - `x_agent`

### 2.2 工具调用语义分离

需要区分两种场景：

#### 场景 A：模型请求外部执行工具
此时可以返回标准 `tool_calls`

适用于：
- 客户端或框架需要接管工具执行
- 兼容 OpenAI tool calling 生态

#### 场景 B：平台后端已自行执行工具
此时**不建议**返回 `tool_calls`
直接返回最终内容，并将过程放入 `x_agent`

适用于：
- Coze / Agent / MCP / Workflow 已在后端跑完
- 前端只负责展示，不负责继续调工具

### 2.3 流式与非流式语义统一

- 非流式：一次性返回最终结果 + 完整扩展信息
- 流式：正文按 OpenAI chunk 增量输出，过程按 `x_agent` 事件输出

### 2.4 跨客户端兼容：严格 OpenAI 流式 vs 扩展流式

**非流式** `POST` 一次性 JSON：多数 SDK / 反序列化会**保留并忽略**未知顶层键（如 `x_agent`），兼容面较宽。客户端若只要标准字段，不读 `x_agent` 即可。

**流式** `text/event-stream` 下，**每一行** `data: {JSON}` 会被部分客户端**单独解析并做 Schema 校验**（例如只接受 `object === "chat.completion.chunk"` 且含 `choices`，或 `error` 对象）。此类实现有时称为 **「严格 OpenAI 流」**。

若在**同一** SSE 流中混插 `object: "agent.event"`（本协议 **§4.2 类型 2**），则：

- **支持扩展的前端/自研监控**：可并行解析两类 `data` 行，展示步骤、状态、推荐问等
- **严格只认 OpenAI 的客户端**（如部分桌面、IDE 类 Chat 应用）：可能抛出 **类型/Union 校验错误**（例如 `invalid_union`、`choices` 缺失）

**落地约定（多项目/网关可统一采用）：**

| 能力 | 说明 |
|------|------|
| **请求体 `full_response`** | 见 **§12.2**，**默认 `false`**：在 JSON 中显式打开「完整返回」时再接扩展，便于同一 Base URL 下普通客户端不改请求、高级客户端设 `true` |
| **环境或配置项** | 用开关控制是否输出流式 `agent.event` 与非流式 `x_agent`（如 `X_AGENT_PROTOCOL=0/1`），**与通用 OpenAI 工具对接时建议默认关** |
| **按请求覆盖** | 对无法改配置的客户端，可用 HTTP 头显式关/开（如 `X-Coze-X-Agent: 0` 强制纯 OpenAI 流；`1` 强制带扩展，具体头名可随项目重命名，但**应在网关文档中写死**） |
| **双通道/双域名** | 大团队也可拆成两个 Base URL：一个**纯标准**、一个**全量扩展**，从路由上消歧义 |

> **与 §5 时序图的关系**：下列「先 thought 再出字」为**推荐叙事顺序**；真实上游（如多轮插件、多段 `message`）可能**在正文 token 与 `agent.event` 之间多次交错**，见 **§5.3**。

## 4. 流式响应协议

## 4.1 设计原则

流式响应继续使用 SSE：

```
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
```

每条消息统一使用：

```
data: {...}
```

结束标记：

```
data: [DONE]
```

## 4.2 流式消息分为两类

### 类型 1：标准 OpenAI 文本 chunk

用于正文输出，尽量兼容 OpenAI 客户端

```
{
  "id": "chatcmpl_001",
  "object": "chat.completion.chunk",
  "created": 1710000000,
  "model": "agent-proxy-v1",
  "choices": [
    {
      "index": 0,
      "delta": {
        "content": "深圳今天多云，"
      },
      "finish_reason": null
    }
  ]
}
```

### 类型 2：平台扩展事件 chunk

用于步骤、状态、确认、推荐问题、UI 信息等

```
{
  "id": "evt_001",
  "object": "agent.event",
  "created": 1710000000,
  "x_agent": {
    "type": "tool_start",
    "step_id": "s2",
    "name": "get_weather",
    "label": "查询天气",
    "status": "running"
  }
}
```

与 **类型 1** 的混排、顺序、以及**正文只拼 `delta.content`** 等消费规则，见 **§2.4、§5.3**。

## 5. 推荐的流式时序

## 5.1 典型流程

```
1. assistant role 开始
2. thought / status 事件
3. tool_start 事件
4. tool_end 事件
5. content 增量输出
6. suggestions / actions 事件
7. stop
8. [DONE]
```

## 5.2 示例

```
data: {"id":"chatcmpl_001","object":"chat.completion.chunk","created":1710000000,"model":"agent-proxy-v1","choices":[{"index":0,"delta":{"role":"assistant"},"finish_reason":null}]}

data: {"id":"evt_001","object":"agent.event","created":1710000001,"x_agent":{"type":"thought","step_id":"s1","message":"需要先查询天气"}}

data: {"id":"evt_002","object":"agent.event","created":1710000002,"x_agent":{"type":"tool_start","step_id":"s2","name":"get_weather","label":"查询天气","status":"running"}}

data: {"id":"evt_003","object":"agent.event","created":1710000003,"x_agent":{"type":"tool_end","step_id":"s2","name":"get_weather","label":"查询天气","status":"success"}}

data: {"id":"chatcmpl_001","object":"chat.completion.chunk","created":1710000004,"model":"agent-proxy-v1","choices":[{"index":0,"delta":{"content":"深圳今天多云，"},"finish_reason":null}]}

data: {"id":"chatcmpl_001","object":"chat.completion.chunk","created":1710000005,"model":"agent-proxy-v1","choices":[{"index":0,"delta":{"content":"25 到 29 度。"},"finish_reason":null}]}

data: {"id":"evt_004","object":"agent.event","created":1710000006,"x_agent":{"type":"suggestions","items":["明天会下雨吗？","这周末适合出门吗？","顺便查一下空气质量"],"show_suggestions":true}}

data: {"id":"chatcmpl_001","object":"chat.completion.chunk","created":1710000007,"model":"agent-proxy-v1","choices":[{"index":0,"delta":{},"finish_reason":"stop"}]}

data: [DONE]
```

## 5.3 流式交错、正文拼接与去重

真实网关对接 **Coze / 工作流 / 多段 assistant 消息** 时，**同一轮回复**的 SSE 往往呈 **多类型交错**（与 §5.1 的理想线性顺序可能不一致），本协议**允许**在时间上任意交错以下两类行（以实际到达顺序为准）：

- `object: "chat.completion.chunk"`
- `object: "agent.event"`

**最终展示给 C 端用户的一段「助手正文」**（产品意义上的 answer），建议**只**通过下列方式得到，**避免重复计算**：

1. **以 OpenAI 为准**：按序拼接所有 `chat.completion.chunk` 中 `choices[0].delta.content` 非空片段（与官方 OpenAI 流式用法一致）。
2. **不要用 `agent.event` 里的整段 `content` 再拼一次**：`agent.event` / `x_agent` 中来自上游 `message.completed` 的摘要常含 **完整或截断的 `type: answer` 文本**，与 **delta 流** 在语义上**同源**，并拼会导致重复或长短不一致。

可选策略（由产品决定，不写入协议硬约束）：

- **隐式推理 / 链式思考**若出现在 `delta.content` 中（如被标签包裹），是否在 UI 中折叠，由客户端**自行剥离**，协议不规定标签名。

**何时读 `agent.event`：**

- 展示 **执行步骤、工具、确认、推荐追问、业务 `status_detail`** 等
- 调试与对账，而非替代 **delta 合成最终 Markdown 正文**

## 6. `x_agent` 统一设计

建议所有扩展信息统一归到 `x_agent`，避免污染标准字段。

## 6.1 顶层结构

```
{
  "x_agent": {
    "trace_id": "trace_xxx",
    "conversation_id": "conv_xxx",
    "run_id": "run_xxx",
    "mode": "agent",
    "status": "running",
    "status_detail": {
      "code": "coze.workflow.node",
      "label": "工作流：审批节点"
    },
    "events": [],
    "ui": {},
    "state": {},
    "meta": {}
  }
}
```

`status_detail` 可缺省；用于承载**平台/业务细粒度**状态（与下方「status」扩展开约定一致）。

## 6.2 字段说明

### trace_id

用于日志追踪与排障

### conversation_id

平台内部会话 ID，可映射 Coze conversation/chat id

### run_id

一次运行的唯一 ID

### mode

当前响应模式，可选：

- `llm`
- `agent`
- `workflow`
- `mcp`

### status

**平台级粗粒度**执行状态，推荐取值（便于监控聚合、通用 UI、与其他系统对齐）：

- `queued`
- `running`
- `waiting_confirm`
- `paused`
- `completed`
- `failed`
- `canceled`

在实现上允许 **额外字符串值**（与接入方/版本号事先约定）作为扩展；**不属于上述集合时**，建议仍配合下方 `status_detail` 或 `events`，避免仅靠单一字符串表达复杂状态机。

### status_detail

可选。承载**细粒度、业务或厂商特有**执行状态，与 `status` 同时出现时：

- **`status`**：仍表示上述**粗粒度**阶段（或你们约定的扩展粗粒度值）
- **`status_detail`**：机器可读的 `code`、人可读的 `label`、以及可选 `raw`（上游原样，如 Coze 事件名、工作流节点 id、错误子码等）

推荐结构：

```
{
  "code": "coze.workflow.node",
  "label": "工作流：审批节点",
  "raw": { "node_id": "n_1", "event": "conversation.xxx" }
}
```

字段说明：

- `code`：稳定、可参与埋点/告警路由（建议点分或 snake_case，如 `acme.billing.hold`）
- `label`：展示用短文案
- `raw`：任意 JSON，**勿**在标准 `message` 里再嵌一层转义 JSON

**流式**：若某轮仅输出 `agent.event`、且需要与最后一帧非流式对齐，可在**结束帧**的 `x_agent` 中带上与聚合结果一致的 `status` / `status_detail`。

### 自定义状态还可以从哪里返回

不破坏 OpenAI 主干的前提下，任选或组合使用：

1. 本节的 **`status_detail`**
2. **`x_agent.events`**：复用 `type: "status"`，在 `message`、事件内 `status` 字符串、`extra` 中写细粒度状态
3. **自定义 `events[].type`**：如 `coze_node_start`、`mcp_invoked`：见 §7.3

标准 OpenAI 客户端可忽略；自有前端、网关、可观测性可订阅上述扩展。

## 7. `x_agent.events` 事件设计

## 7.1 通用事件结构

```
{
  "type": "tool_start",
  "step_id": "s1",
  "name": "get_weather",
  "label": "查询天气",
  "status": "running",
  "message": "正在查询天气",
  "extra": {}
}
```

## 7.2 推荐事件类型

### thought

表示模型/Agent 的中间思考摘要

```
{
  "type": "thought",
  "step_id": "s1",
  "message": "需要先查询天气"
}
```

### status

表示普通状态变化

```
{
  "type": "status",
  "step_id": "s1",
  "status": "running",
  "message": "正在处理请求"
}
```

### tool_start

工具开始执行

```
{
  "type": "tool_start",
  "step_id": "s2",
  "name": "get_weather",
  "label": "查询天气",
  "status": "running"
}
```

### tool_end

工具执行完成

```
{
  "type": "tool_end",
  "step_id": "s2",
  "name": "get_weather",
  "label": "查询天气",
  "status": "success"
}
```

### tool_error

工具执行失败

```
{
  "type": "tool_error",
  "step_id": "s2",
  "name": "get_weather",
  "label": "查询天气",
  "status": "failed",
  "message": "天气服务不可用"
}
```

### knowledge_hit

命中知识库

```
{
  "type": "knowledge_hit",
  "step_id": "s3",
  "message": "已命中知识库",
  "extra": {
    "source_count": 3
  }
}
```

### waiting_confirm

进入等待确认状态

```
{
  "type": "waiting_confirm",
  "step_id": "s4",
  "message": "删除文件前需要确认",
  "confirm_key": "confirm_123",
  "actions": [
    {"key": "approve", "label": "确认"},
    {"key": "reject", "label": "取消"}
  ]
}
```

### resume

确认后继续执行

```
{
  "type": "resume",
  "step_id": "s4",
  "message": "已确认，继续执行"
}
```

### suggestions

推荐追问

```
{
  "type": "suggestions",
  "show_suggestions": true,
  "items": [
    "明天会下雨吗？",
    "这周末适合出门吗？"
  ]
}
```

### actions

推荐操作

```
{
  "type": "actions",
  "items": [
    {"key": "web_search", "label": "联网搜索", "type": "action"},
    {"key": "generate_report", "label": "生成报告", "type": "action"}
  ]
}
```

### done

流程结束事件

```
{
  "type": "done",
  "status": "completed"
}
```

## 7.3 自定义事件类型与 `extra` 约定

上文 **§7.2 推荐事件类型** 为**最小可互操作集**；`type` 保持**可扩展**（见 §15.2）。接入 Coze、MCP、自研工作流时，可新增业务类型，不占用 OpenAI 标准字段。

**命名**（建议）：

- 小写 + 下划线；需要避免冲突时加**业务前缀**，如 `acme_approval_required`、`coze_workflow_step`
- 与 HTTP 或内部子系统枚举对齐时，优先用**稳定**的 `code` 放在 `extra.code`，`type` 仍用语义化英文

**载荷**：

- 结构化数据、上游透传、节点 id、重试次数等，放在 **`extra`**；避免把大段 JSON 拼进 `message`
- 若需表达「子状态机」，推荐：`type: "status"` + `extra: { "phase": "retrying", "attempt": 2 }` 或 **自定义 `type` + `extra`**

**流式**（`object: "agent.event"` 的 `x_agent` 内层）与 **非流式**（`x_agent.events[]`）使用**相同**的 `type` / `extra` 约定，便于同一套前端逻辑解析。

**示例（自定义 + extra）**：

```
{
  "type": "coze_workflow_step",
  "step_id": "s9",
  "message": "进入审批子流程",
  "extra": {
    "workflow_id": "wf_1",
    "node_key": "approval",
    "coze": { "bot_id": "7628..." }
  }
}
```

## 8. `x_agent.ui` 设计

用于承载前端展示增强信息，避免把 UI 数据混在 `events` 中。

## 8.1 示例结构

```
{
  "ui": {
    "show_suggestions": true,
    "suggestions": [
      "继续追问",
      "查看详情"
    ],
    "actions": [
      {
        "key": "ask_followup",
        "label": "继续追问",
        "type": "followup"
      },
      {
        "key": "web_search",
        "label": "联网搜索",
        "type": "action"
      }
    ],
    "badges": [
      {
        "key": "knowledge_hit",
        "label": "知识库命中"
      }
    ]
  }
}
```

## 8.2 建议字段

### show_suggestions

是否展示推荐问题

### suggestions

推荐问题数组

### actions

可执行按钮数组

### badges

状态徽标，如：

- 知识库命中
- 工具执行
- 需确认
- 高可信

## 11. 是否使用 `tool_calls`

## 11.1 建议原则

### 使用 `tool_calls`

适用于：

- 需要兼容 OpenAI 工具调用框架
- 需要由客户端或中间框架继续执行工具

### 不使用 `tool_calls`

适用于：

- 工具已由后端自动执行
- 前端只展示最终结果
- 当前响应本质是 Agent 执行结果而非“请求调工具”

## 11.2 平台建议

平台默认建议：

- **默认模式**：不返回 `tool_calls`
- **兼容模式**：按需返回 `tool_calls`

可通过请求参数控制：

```
{
  "tool_call_mode": "none"
}
```

可选值：

- `none`：不返回 `tool_calls`
- `compat`：尽量兼容 OpenAI tool calling
- `native`：平台内部原生 Agent 表达

## 12. 请求侧建议扩展

为了适配你的平台，建议请求体也允许扩展字段。

## 12.1 示例

```
{
  "model": "agent-proxy-v1",
  "messages": [
    {
      "role": "user",
      "content": "帮我查一下深圳天气"
    }
  ],
  "stream": true,
  "full_response": false,
  "x_agent": {
    "mode": "agent",
    "tool_call_mode": "none",
    "show_suggestions": true,
    "enable_events": true,
    "conversation_id": "conv_xxx"
  }
}
```

- `full_response` 也可放在与 `x_agent` 同级的**顶层**（上例已示）；若实现将扩展开关放在 `x_agent` 内，则等价字段可命名为 `x_agent.full_response`（**二选一，勿重复**）。

## 12.2 推荐字段

### full_response

- 类型：`boolean`
- 默认： **`false`**
- 含义：是否走「**完整**」统一协议能力——**非流式**在 JSON 中带顶层 **`x_agent`**、**流式**在 SSE 中输出 **`agent.event`** 等过程信息（与 **§2.4**、**§5.3** 一致）。  
- **`false`**：响应形态与**纯 OpenAI** 尽量对齐，普通客户端可不做 Union 白名单。  
- **`true`**：输出扩展；可与网关的**环境变量 / 请求头**（如 `X-Coze-X-Agent`）做组合，**优先级建议在 OpenAPI 中写死**（常见：请求体 > 请求头 > 环境默认）。

### mode

执行模式：

- `llm`
- `agent`
- `workflow`

### tool_call_mode

是否输出 `tool_calls`

### show_suggestions

是否返回推荐问题

### enable_events

流式时是否输出 `agent.event`。若已实现顶层 **`full_response`**，建议**仅当** `full_response === true` 且 `stream === true` 时本字段才生效；也可省略，仅用 `full_response` 作总闸。

### conversation_id

外部传入会话 ID

## 13. 前端消费建议

## 13.1 普通 OpenAI 客户端

只处理：

- `choices[].message.content`（非流式）
- `choices[].delta.content`（流式，按时间顺序拼接，见 **§5.3** 避免与扩展重复）
- `[DONE]`

不读取（非流式下可**丢在 JSON 里**不解析）：

- 顶层 `x_agent`

不读取，且**流式下若出现则会校验失败**（需网关侧**关闭扩展**或换 Base URL，见 **§2.4**）：

- `data:` 行里 `object: "agent.event"`

## 13.2 平台自有客户端

额外处理：

- `x_agent.events`
- `x_agent.ui`
- `x_agent.state`
- `x_agent.meta`

### 推荐渲染规则

#### 非流式

- 直接展示最终文本
- 根据 `x_agent.events` 渲染步骤列表
- 根据 `x_agent.ui.suggestions` 渲染推荐追问
- 根据 `x_agent.ui.actions` 渲染操作按钮

#### 流式

- `chat.completion.chunk`：拼接正文
- `agent.event`：更新步骤 UI、状态 UI、推荐问题 UI

## 17. 多项目复用与网关实现要点

其它项目若复用本协议，建议在实现层**显式**处理下列事项，减少与不同客户端的隐性冲突（参考实现思路：环境变量 + 每请求头覆盖 + 流式/非流式分支）。

### 17.1 能力开关

| 维度 | 建议 |
|------|------|
| 请求体 `full_response` | 建议支持，**默认 `false`**，与纯 OpenAI 工具对接时不必改请求即可关闭扩展（见 **§12.2**） |
| 非流式 `x_agent` | 可独立开关；关闭时响应与纯 OpenAI `chat.completion` 一致 |
| 流式 `agent.event` | **强烈建议**独立开关，且对接「只认 `choices`」的客户端时默认**关闭**（见 **§2.4**） |
| 流式行数上限 | 对 `agent.event` 设单请求上限，避免长工作流刷爆弱客户端与带宽 |

### 17.2 与上游的映射

- 上游（如 **Coze Chat V3**）以 **SSE `event` 名 + `data` JSON** 送达；若映射为本协议的 `coze_message` / `coze_sse` 等，应对 **payload 做脱敏/截断**（长 `content`、内嵌 JSON 字符串、URL 查询串等），`extra` 中保留**可观测、可路由**的键即可。
- **`ping` / 心跳**类事件可映射为轻量 `agent.event` 或**直接忽略**（由网关定策略）。

### 17.3 与 OpenAI 主干的 ID 与结束语义

- `chatcmpl-…` 与 `evt-…` **id 空间分离**，避免客户端误把扩展行当成同一条 completion 的 chunk。
- 最后一帧仍为 **`finish_reason: "stop"` 的 `chat.completion.chunk`**（`usage` 按你们与 OpenAI 的对齐方式放在 `chunk` 内或末包），紧接 `data: [DONE]`。

### 17.4 复制到其它仓库时的「最小文档包」

向协作方或另一仓库同步时，建议**至少**带走：

- **§1–2**（目标与原则，含 **§2.4**）
- **§3–4**（非流式 / 流式基本形状）
- **§5.3**（流式交错与**禁止双拼正文**）
- **§6–7**（`x_agent` 与 `events` 扩展）
- **§17**（本实现要点）

并在外层项目的 **OpenAPI / 网关 README** 中写清：扩展开关、请求头、以及「严格 OpenAI 客户端请关扩展」**一句话**。

---

## 18. 最终结论

本协议建议采用：

> **OpenAI 主干兼容 + `x_agent` / `agent.event` 扩展承载 Agent 能力**；**流式扩展与严格 OpenAI 客户端通过开关与 §5.3 的拼接规则解耦**

这样可以同时满足：

- 在关闭扩展或纯 OpenAI 路由时，**尽量兼容**现有只认 `choices` 的客户端
- 在开启扩展时，保留工具流、步骤流、状态与**可截断的**原始轨迹
- 支持自研前端的更富交互
- 多项目复用时，用 **§17** 的配置与文档约定**统一踩坑面**
- 降低后续接入 Coze、MCP、Workflow 的协议改造成本
